# Internationalization (i18n)

Coat Check ships English (`en`, the source/fallback) and Norwegian Bokmål (`nb`). The system is
deliberately lightweight — no i18n routing, no dependency.

## How it works

- **Locale is a per-account preference, not a URL.** Signed-in users have `accounts.locale`
  (migration `0011`); signed-out users follow their browser language. No `/en/` `/nb/` paths.
- **Resolution + persistence:** `components/MeProvider.tsx` fetches `/api/me` once;
  `components/I18nProvider.tsx` reads `me.account.locale` (falling back to the browser language)
  and exposes `useT()` / `useI18n()`. Changing the switcher calls `PUT /api/locale`, so the choice
  follows the user to every device (and future native clients can read the same preference).
- **Messages** live in `messages/en.ts` and `messages/nb.ts` as nested dictionaries. `en.ts` is the
  canonical shape (`export type Messages = typeof en`); every other locale is typed `Messages`, so
  **a missing key is a compile error**.
- **Lookup:** `useT()` returns `t('a.b.c', { var })` — dot-pathed, with `{var}` interpolation and an
  English fallback for any untranslated key. Two domain helpers: `itemName(t, item)` (clothing by
  catalog id) and `weatherName(t, code, fallback)` (WMO code), both in `lib/i18n.ts`.
- **Dates/numbers** format via `Intl` using `LOCALE_TAG` (`lib/i18n.ts`) — free for any locale.

## Adding a language

Translation strings are the only real work; the plumbing is ~5 one-line edits. To add e.g. Spanish
(`es`):

1. **`messages/es.ts`** — copy `messages/nb.ts`, change the import to
   `import type { Messages } from './en'`, and translate every value. TypeScript will refuse to
   compile until all keys are present (this is your completeness checklist).
2. **`lib/i18n.ts`** — five edits:
   - `export type Locale = 'en' | 'nb' | 'es';`
   - `export const LOCALES = ['en', 'nb', 'es'];`
   - `import { es } from '@/messages/es'` + add it to `MESSAGES`.
   - `localeFromBrowser`: map the browser tag (e.g. `if (l.startsWith('es')) return 'es';`).
   - `LOCALE_TAG`: add `es: 'es-ES'`.
3. **Add a short display label** for the switcher. The header switcher
   (`app/page.tsx`) renders one `<option>` per `LOCALES` entry automatically (label = the code
   uppercased), so **no UI edit is needed** unless you want a prettier label than `ES`.

That's it — every translated string updates everywhere `useT()` is used, dates reformat via `Intl`,
and clothing/weather translate by id/code.

## What's translated

The home page (chrome, search, scene, week strip, list view, comfort picker, exploded outfit),
clothing names, weather descriptions, the Family page, and the Trips pages. **Not yet translated**
(English-only, low-traffic): the Beta and Admin pages, and a handful of error-fallback strings.
Add `beta.*` / `admin.*` keys and wire them the same way when needed.

## Key files

| File | Role |
|---|---|
| `lib/i18n.ts` | `Locale`, `LOCALES`, `LOCALE_TAG`, `makeT`, `itemName`, `weatherName` |
| `messages/{en,nb}.ts` | the dictionaries (`en` is the canonical shape) |
| `components/MeProvider.tsx` | single `/api/me` fetch (locale source) |
| `components/I18nProvider.tsx` | locale resolution + `useT()` / `useI18n()` |
| `app/api/locale/route.ts` | persists `accounts.locale` |
